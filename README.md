# RAG-ассистент по регламентам и правовым документам

ИИ-ассистент с RAG (Retrieval-Augmented Generation): отвечает на вопросы по внутренним регламентам, политикам и правовым документам. Использует ChromaDB, OpenAI API (эмбеддинги и LLM) и кеширование ответов в SQLite. Запускается из терминала: `python app.py`.

## Описание

Система помогает быстро находить нужные формулировки и разделы в объёмных регламентах и правовых текстах. Ассистент не заменяет юриста или ответственного сотрудника, но ускоряет поиск по базе знаний.

**Как это работает:**

1. Пользователь задаёт вопрос.
2. Проверка кеша — если ответ уже есть, возвращается из кеша (SQLite).
3. Векторный поиск в ChromaDB по релевантным фрагментам (чанкам).
4. Найденные чанки и вопрос передаются в LLM (OpenAI).
5. Ответ кешируется и возвращается пользователю.

## Документы для базы знаний (папка `data/`)

В векторную базу попадают **все .txt файлы** из папки **`data/`** (ГПК, АПК, ГК РФ, претензия и др.). Для каждого файла задаётся кодекс/тип для фильтрации запросов.

### Какие документы подойдут

| Тип | Примеры |
|-----|--------|
| Внутренние регламенты | Регламент работы с клиентами, регламент согласования, регламент отпусков |
| Политики и правила | Политика конфиденциальности, правила внутреннего распорядка, кодекс этики |
| Правовые выдержки | Статьи законов, приказы, инструкции (ассистент помогает найти формулировки, не заменяя юриста) |
| Инструкции и шаблоны | Порядок оформления заявки, порядок подачи жалобы, типовые формулировки |

### Формат

- Поддерживается **текст в кодировке UTF-8**. Все **.txt** из папки `data/` загружаются автоматически; в ответе и в логах выводятся источник (документ) и кодекс.
- В `.env` можно задать **TOP_K** — сколько чанков передавать в LLM (по умолчанию 5).
- PDF/DOCX нужно конвертировать в .txt и положить в `data/`.

### Фильтрация по кодексам

Поиск сужается до нужного кодекса или типа документа, чтобы не смешивать, например, ГПК и АПК.

**Как это устроено:**

1. **При загрузке документов** (`vector_store.py`): у каждого файла из `data/` задаётся метаданные — **source** (имя файла без .txt) и **code** (кодекс/тип). Соответствие имя файла → код задаётся в `FILE_TO_CODE` (например: `gpk_rf` → «ГПК РФ», `ПРЕТЕНЗИЯ` → «претензия»). Все чанки из этого файла сохраняются в ChromaDB с полями `source` и `code`.

2. **При вопросе пользователя** (`app.py`, `bot_telegram.py`): по тексту вопроса определяется фильтр. Словарь **QUERY_TO_FILTER** сопоставляет ключевые слова с кодексом: «ГПК», «гражданск», «мировой суд» → «ГПК РФ»; «АПК», «арбитраж» → «АПК РФ»; «претензи» → «претензия»; «ГК РФ», «гражданский кодекс» → «ГК РФ». Функция `_infer_filter(query)` возвращает, например, `{"code": "ГПК РФ"}` или `None`, если подходящих слов нет.

3. **При поиске в ChromaDB**: если фильтр задан, в запрос к векторному хранилищу передаётся `where=metadata_filter` (например `where={"code": "ГПК РФ"}`). В результат попадают только чанки с этим кодексом; семантический поиск идёт уже внутри подмножества.

**Итог:** вопрос «Как подать иск по ГПК?» → фильтр «ГПК РФ» → поиск только по чанкам из документов ГПК; ответ строится по релевантным фрагментам именно этих документов. Если в вопросе нет ключевых слов, фильтр не ставится и поиск идёт по всей базе.

## Установка

Рекомендуется **Python 3.11** (с 3.13 возможны конфликты зависимостей, в т.ч. с RAGAS).

```bash
# Переход в директорию проекта
cd "RAG-ассистент по регламентам и правовым документам"

# Виртуальное окружение (рекомендуется)
python -m venv venv
# Windows (PowerShell):
.\venv\Scripts\Activate.ps1
# Windows (cmd) / Linux / Mac:
# venv\Scripts\activate   или   source venv/bin/activate

# Зависимости
pip install -r requirements.txt
```

### Настройка OpenAI API

Создайте в корне проекта файл `.env`:

```env
OPENAI_API_KEY=your_openai_api_key_here
TOP_K=5
# Для Telegram-бота (python bot_telegram.py):
# TELEGRAM_BOT_TOKEN=123456:ABC-...
```
(TOP_K — сколько чанков передавать в LLM; по умолчанию 5.)

Ключ можно получить на [platform.openai.com](https://platform.openai.com/api-keys).

## Запуск

```bash
python app.py
```

При первом запуске:

- Создаётся директория ChromaDB (`chroma_db/`).
- Документы из папки `data/` (все .txt) разбиваются на чанки, для них создаются эмбеддинги (OpenAI) и записываются в ChromaDB.
- Далее можно задавать вопросы в консоли.

**Команды в чате:** `exit` / `quit` — выход, `stats` — статистика, `clear` — очистка кеша, `tests` — показать вопросы из файла.

### Запуск Telegram-бота

Вопросы можно задавать в Telegram:

1. Создайте бота через [@BotFather](https://t.me/BotFather), получите токен.
2. В `.env` добавьте строку: `TELEGRAM_BOT_TOKEN=ваш_токен`.
3. Установите зависимость: `pip install python-telegram-bot`
4. Запустите: `python bot_telegram.py`

В чате с ботом напишите вопрос — он ответит по той же RAG-базе (общий кеш с консольным режимом).

## Команды в боте: анализ ответов

Две команды для проверки качества ответов LLM через RAGAS:

| Команда | Что анализируется | Как использовать |
|--------|--------------------|------------------|
| **`analyze`** | **Последний ответ в боте** (один вопрос → один ответ) | Задайте вопрос, получите ответ, затем введите **`analyze`** — RAGAS оценит именно этот ответ и контекст (Faithfulness, Context Precision). |
| **`eval`** | **Ответы по всем вопросам из файла** | Введите **`eval`** — бот по очереди «отвечает» на каждый вопрос из `evaluation_questions.txt`, затем RAGAS считает метрики по всем этим ответам и выводит средние баллы. |

- **Анализ ответа в боте:** задать вопрос → прочитать ответ → ввести **`analyze`**.
- **Анализ по файлу:** положить вопросы в `evaluation_questions.txt` (один на строку) → в боте ввести **`eval`**.

## Оценка качества (RAGAS)

Оценку можно запустить **из бота** (команда **`eval`**) или **отдельным скриптом**:

```bash
python evaluate_ragas.py
```

Скрипт читает вопросы из `evaluation_questions.txt` (один на строку), для каждого получает ответ через RAG и передаёт в RAGAS. Рекомендуется Python 3.11.

# Быстрые тесты (файл вопросов, _infer_filter) — без API

## Архитектура системы

Пайплайн: **ingest → retrieval → фильтр → prompt → LLM → кеш → метрики**. RAG не запоминает документы, а каждый раз ищет релевантные фрагменты и генерирует ответ по контексту.

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  ИСТОЧНИК ДАННЫХ                    ПРЕПРОЦЕССИНГ              ЭМБЕДДИНГИ             │
│  data/*.txt (ГПК, АПК, претензия) → разбиение на чанки,   →   векторы (OpenAI)          │
│  Документы, регламенты, FAQ           overlap ~20%                                         │
└─────────────────────────────────────────────────────────────────────────────────────────┘
                                                    │
                                                    ▼
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  ВЕКТОРНОЕ ХРАНИЛИЩЕ (ChromaDB)                                                         │
│  Хранение и быстрый поиск по смыслу. Индексация при первом запуске.                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘
                                                    │
         Запрос пользователя                         │  Поиск (semantic search, top_k)
         (консоль / Telegram)                        ▼
┌──────────────────┐     кеш есть?     ┌─────────────────────────────────────────────────┐
│  КЕШ (SQLite)    │ ◄───────────────── │  Поиск релевантных чанков + фильтр по кодексу   │
│  вопрос → ответ  │   да: ответ из     │  (ГПК РФ, АПК РФ, претензия и т.д.)              │
└────────┬─────────┘   кеша            └─────────────────────────────────────────────────┘
         │ нет                                                      │
         ▼                                                          ▼
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  ПРОМПТ (контекст + вопрос)  →  LLM (OpenAI gpt-4o-mini)  →  Ответ  →  сохранение в кеш │
└─────────────────────────────────────────────────────────────────────────────────────────┘
                                                    │
                                                    ▼
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  ОТВЕТ ПОЛЬЗОВАТЕЛЮ (с указанием источников). При необходимости — RAGAS (Faithfulness,   │
│  Context Precision) через команды analyze / eval или скрипт evaluate_ragas.py.           │
└─────────────────────────────────────────────────────────────────────────────────────────┘
```

**Компоненты:**

| Компонент | Файл / место | Назначение |
|-----------|---------------|------------|
| Загрузка и чанкинг | `vector_store.py` | Чтение .txt из `data/`, разбиение на чанки (~500 символов, overlap), эмбеддинги, запись в ChromaDB |
| Векторное хранилище | ChromaDB (`chroma_db/`) | Поиск по смыслу (cosine similarity), фильтр по полю «кодекс» |
| Кеш | `cache.py`, SQLite | Повторные запросы без вызова API; в ответе — «Источник: КЕШ» |
| Пайплайн | `rag_pipeline.py` | Кеш → поиск → фильтр по кодексу → промпт → LLM → кеш; формы претензии/иска — дословно из файлов |
| Точки входа | `app.py`, `bot_telegram.py` | Консоль и Telegram; общий кеш и векторная база |
| Оценка качества | RAGAS (`analyze`, `eval`, `evaluate_ragas.py`) | Faithfulness, Context Precision, средний балл по вопросам |

## Как масштабировать

Кратко — где и что менять, чтобы адаптировать систему под другие объёмы или стек.

| Задача | Где менять | Что сделать |
|--------|------------|-------------|
| **Поменять модель LLM** | `rag_pipeline.py` | В вызове OpenAI задать другую `model` (например `gpt-4o`, `gpt-4-turbo`). Учесть лимиты токенов и стоимость. |
| **Изменить число чанков (top_k)** | `.env` или код | В `.env`: `TOP_K=7` (или другое значение). В коде: параметр `top_k` в `rag_pipeline.py` при запросе к векторному хранилищу. |
| **Добавить новый кодекс/тип документа** | `data/` + метаданные | Положить в `data/` новый .txt. В `vector_store.py` при загрузке задаётся метаданные (документ, кодекс). В `rag_pipeline.py` логика фильтра по ключевым словам (ГПК, АПК, претензия и т.д.) — при необходимости добавить новый ключ. |
| **Заменить Chroma на другое векторное хранилище** | `vector_store.py` | Реализовать тот же интерфейс: загрузка документов с метаданными, метод поиска с фильтром и `top_k`. Подключить, например, Pinecone, Weaviate, Qdrant или pgvector; эмбеддинги можно оставить OpenAI или перейти на локальные. |

После смены модели или хранилища имеет смысл перезапустить оценку RAGAS (`evaluate_ragas.py` или команда `eval`) и при необходимости очистить кеш (`clear`).

## Технические детали

- **Эмбеддинги:** OpenAI `text-embedding-3-small` (1536 размерность).
- **LLM:** по умолчанию `gpt-4o-mini` (задаётся в `rag_pipeline.py`).
- **Чанкинг:** целевой размер ~500 символов, перекрытие ~100 символов, разбиение по абзацам и предложениям.
- **ChromaDB:** метрика cosine similarity, данные в `./chroma_db/`.
- **Кеш:** SQLite, ключ — хеш запроса.

## Ограничения

- Нужен действующий API ключ OpenAI (платные запросы на эмбеддинги и генерацию).
- При обновлении документов в `data/` нужно либо пересоздать коллекцию ChromaDB (удалить `chroma_db/` и запустить снова), либо доработать код переиндексации; кеш при смене документов лучше очищать командой `clear`.

## Возможное развитие (next-level)

Шаг в сторону production RAG — **логирование запросов** для последующего анализа:

- Сохранять в БД или файл: вопрос, ответ, использованные чанки, модель, число токенов (если API возвращает usage).
- На основе логов анализировать качество ответов, частые запросы, стоимость токенов и донастраивать top_k, чанкинг или промпты.

Реализация: отдельный модуль или обёртка вокруг `rag_pipeline.answer()`, запись в SQLite/JSON после каждого ответа; при необходимости — дашборд или отчёт по логам.

## Тестирование и сдача ДЗ
