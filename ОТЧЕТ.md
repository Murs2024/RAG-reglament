Система запускается из терминала (или по **Ctrl+Shift+B** в Cursor). При старте выводятся:

- **Инициализация кеша** (SQLite) — кеш готов к сохранению и выдаче ответов; при первом запросе ответ сохраняется в кеш, при повторе того же вопроса выдаётся «Источник: КЕШ» (проверка в п. 2).

## 2. Первый запрос и проверка кеша (повтор того же вопроса)

**Вопрос:** «Как составить исковое заявление по ГПК?»

**Первый раз (ответ из API):** проверка кеша → «В кеше нет → идём в векторную БД» → поиск 5 чанков (gpk_rf, ГПК РФ) → промпт → генерация через OpenAI API (gpt-4o-mini) → ответ получен → **сохранение в кеш**. В выводе: Источник: **OpenAI API (gpt-4o-mini)**, использовано документов: 5; текст ответа (требования к исковому заявлению, ст. 131 ГПК РФ); блоки источников по чанкам и превью фрагментов.

**Повторно (ответ из кеша):** тот же вопрос задаётся ещё раз. В выводе: «[*] Проверка кеша…» → «[+] Ответ найден в кеше». Ответ пользователю: **Источник: КЕШ**, **Сохранено:** 2026-03-01T19:48:00.298219; текст ответа тот же, блоки источников (gpk_rf, ГПК РФ) выводятся как при первом запросе.

Итого: в п. 2 проверены и первый запрос (API + сохранение в кеш), и повтор того же вопроса (ответ из кеша).

*Скриншоты: 1) первый запрос — полный лог (кеш нет → БД → API → сохранено в кеш), ответ и источники; 2) повтор — «Ответ найден в кеше», Источник: КЕШ, время сохранения.*

## 3. Запрос по АПК (второй вопрос)

**Вопрос:** «Куда подавать иск по АПК?»

**Ответ пользователю:**

- **Источник:** OpenAI API (gpt-4o-mini), **использовано документов:** 5.
- **Ответ:** Исковое заявление по АПК подаётся в арбитражный суд по месту нахождения имущества (если речь о правах на недвижимость); по искам к перевозчику — по месту нахождения перевозчика; для иных категорий дел действуют специальные правила, как указано в **статьях 38 и 225.1 АПК РФ**. Источник: АПК РФ.

**Источники (по чанкам из векторной БД):**

1. документ: ИСКОВОЕ ЗАЯВЛЕНИЕ ПО АПК РФ, кодекс: АПК РФ  
2. документ: apk_rf, кодекс: АПК РФ  
3. документ: apk_rf, кодекс: АПК РФ  
4. документ: apk_rf, кодекс: АПК РФ  
5. документ: apk_rf, кодекс: АПК РФ  

**Источники (документ / кодекс):** превью фрагментов с указанием ст. 125–126, 38, 93, 148 АПК РФ, раздел «Исключительная подсудность», «ПРОШУ СУД», образцы формулировок иска, примечания КонсультантПлюс.

**Вывод:** по ключевому слову «АПК» сработал фильтр по кодексу АПК РФ; поиск вернул чанки из документов `apk_rf` и «ИСКОВОЕ ЗАЯВЛЕНИЕ ПО АПК РФ»; ответ соответствует арбитражной подсудности и ссылается на статьи АПК РФ.

*Скриншот: вопрос «Куда подавать иск по АПК?», ответ с источниками (АПК РФ, apk_rf, ИСКОВОЕ ЗАЯВЛЕНИЕ ПО АПК РФ), превью фрагментов.*

## 4. Команда stats — статистика системы

В консольном боте введена команда `stats` (или **статистика**).

Получен блок **«СТАТИСТИКА СИСТЕМЫ»**:
- **Векторное хранилище:** коллекция `api_rag_collection`, число документов (чанков), директория (например `./chroma_db`);
- **Кеш:** число записей, размер БД (MB), первая и последняя запись;
- **Модель:** gpt-4o-mini;
- **Чанков в LLM (top_k):** 5;
- **Режим:** API.

Команда stats вывела актуальное состояние векторной БД, кеша и параметров модели.

*Скриншот: «Ваш вопрос: stats» и блок «СТАТИСТИКА СИСТЕМЫ» с векторным хранилищем, кешем, моделью, top_k.*

## 6. Команда analyze — RAGAS по одному ответу

Задан вопрос: «Что должно быть в исковом заявлении по ГПК РФ?»; после ответа с источниками введена команда `analyze`.

Получен блок **«АНАЛИЗ ПОСЛЕДНЕГО ОТВЕТА (RAGAS)»** с метриками:
- **Faithfulness (ответ по контексту):** 1.0000
- **Context Precision (релевантность):** 1.0000

Ход до этого: проверка кеша → «В кеше нет» → поиск 5 чанков (gpk_rf, ГПК РФ) → промпт → API (gpt-4o-mini) → ответ получен → сохранение в кеш. Ответ содержит 8 пунктов требований к исковому заявлению (ст. 131, 132 ГПК РФ). RAGAS оценивает, насколько ответ соответствует контексту и насколько релевантны найденные чанки — оба показателя 1.0 (максимум).

*Скриншот: «Ваш вопрос: analyze» и блок «АНАЛИЗ ПОСЛЕДНЕГО ОТВЕТА (RAGAS)» с Faithfulness и Context Precision.*

## 7. Команда eval — оценка RAGAS по всем вопросам из файла

В консольном боте введена команда `eval`.

Система запустила оценку качества RAG по метрикам RAGAS: берёт все вопросы из `evaluation_questions.txt` (5 шт.), для каждого получает ответ через RAG (поиск в векторной БД с фильтром по кодексу → промпт → OpenAI API), затем вычисляет Faithfulness и Context Precision по всем парам «вопрос — ответ — контекст».

После прохода по всем вопросам выведен итоговый блок **«МЕТРИКИ»**:
- **Faithfulness (точность ответа):** 0,9467
- **Context Precision (точность контекста):** 0,9775
- **ИТОГО средний балл:** 0,9621
- **Оценка:** отличное качество [OK]

Оценка может быть: [OK] (отличное), удовлетворительно или требуется улучшение — в зависимости от среднего балла.

*Скриншот: блок «МЕТРИКИ» в конце вывода команды eval — Faithfulness, Context Precision, ИТОГО средний балл, оценка.*

## 8. Скрипт evaluate_ragas.py (оценка без консольного бота)

Консольный бот закрыт (`exit`), в терминале выполнен запуск:
```
python evaluate_ragas.py
```

Скрипт инициализировал RAG (векторное хранилище, кеш), загрузил вопросы из `evaluation_questions.txt` (5 шт.), для каждого получил ответ через RAG и посчитал метрики RAGAS. В конце выведены:
- **РЕЗУЛЬТАТЫ ОЦЕНКИ** — средние значения Faithfulness и Context Precision, средний балл, оценка (например «Отличное качество! [OK]»);
- **ДЕТАЛЬНЫЕ РЕЗУЛЬТАТЫ ПО ВОПРОСАМ** — по каждому из 5 вопросов: Faithfulness и Context Precision;
- строка **«Оценка завершена!»** или **«[ОК] Оценка завершена!»**.

Оценка RAG через RAGAS выполнена отдельным скриптом (как и командой `eval` в боте).

*Скриншот: запуск `python evaluate_ragas.py` — блок «РЕЗУЛЬТАТЫ ОЦЕНКИ» и «ДЕТАЛЬНЫЕ РЕЗУЛЬТАТЫ ПО ВОПРОСАМ», сообщение об завершении.*

Telegram-бот корректно обработал запрос о форме претензии, вернул документ из файла целиком (в нескольких сообщениях) и указал источник.
